# Feishu Base 管理后台使用说明

本文档用于规范 `ruc-suzhou-case-library` 当前的飞书后台维护方式。

当前已经打通的链路是：

```text
Feishu Base（offer表 + 页面配置表）
  -> 导出脚本
  -> ruc-suzhou-case-library/src/data/generated/frontendData.js
  -> 前台展示页
```

也就是说：

- `offer表` 管案例内容
- `页面配置表` 管页面文案配置
- 每次你修改后台表后，需要重新运行一次导出脚本

```bash
python3 feishu-bitable-sync/export_frontend_data.py
```

## 1. offer表：推荐字段规范

### 1.1 字段分层

为了避免后台越来越乱，建议把 `offer表` 的字段分成三层理解：

#### A. 必填主展示字段
这些字段决定前台能不能正常显示。

| 字段名 | 是否建议必填 | 用途 | 说明 |
| --- | --- | --- | --- |
| 案例ID | 是 | 唯一标识 | 建议保持稳定，不要频繁修改 |
| 申请季 | 是 | 列表展示 / 筛选 | 例如：`24fall` / `25fall` / `26fall` |
| 录取院校 | 是 | 列表标题 / 详情页 | 当前脚本会优先读 `录取学校`，为空则回退 `录取院校` |
| 录取专业 | 是 | 列表标题 / 详情页 | 建议直接写可对外展示版本 |
| 本科学校 | 是 | 列表 / 详情页 / 筛选 | 当前脚本会回退到 `本科学校` |
| 本科专业 | 是 | 列表 / 详情页 / 筛选 | 建议写完整标准名称 |
| GPA/均分 | 是 | 列表 / 详情页 | 保持原始表达即可 |
| 雅思 / 托福 / GRE/GMAT | 至少一项建议有 | 列表 / 详情页 | 空字段会自动隐藏 |

#### B. 强烈建议填写的前台控制字段
这些字段不是绝对必填，但如果不填，后台控制力会比较弱。

| 字段名 | 是否建议填写 | 用途 | 建议规则 |
| --- | --- | --- | --- |
| 是否前台展示 | 强烈建议 | 控制是否上架 | 建议统一填：`是 / 否` |
| 排序权重 | 强烈建议 | 控制排序 | 数字越大越靠前 |
| 置顶标记 | 建议 | 控制置顶 | 建议统一填：`是 / 否` |
| 匿名展示名 | 强烈建议 | 控制前台学生名 | 例如：`A同学`、`L同学` |
| 录取国家/地区 | 强烈建议 | 用于筛选 | 例如：`中国香港`、`英国`、`新加坡` |
| 标签 | 建议 | 搜索增强 / 后续展示 | 多个标签建议用 `/` 分隔 |

#### C. 详情页增强字段
这些字段决定详情页是否更完整。

| 字段名 | 是否建议填写 | 用途 | 说明 |
| --- | --- | --- | --- |
| 实习经历 | 建议 | 详情页展示 | 可空 |
| 科研经历 | 建议 | 详情页展示 | 可空 |
| 申请轮次 | 建议 | 详情页展示 | 如 round 1 / 提前批 |
| 录取时间 | 建议 | 详情页展示 | 建议统一格式 |
| 备注 | 选填 | 详情页展示 | 列表页不展示，详情页有值才展示 |
| 最终去向说明 | 选填 | 详情页展示 | 用于补充实际去向 |
| 案例说明/项目说明 | 选填 | 搜索 / 详情页补充 | 建议简洁，不要写成长文 |

#### D. 学生名片控制字段
这些字段专门控制详情页里的学生名片模块。

| 字段名 | 是否建议填写 | 用途 | 说明 |
| --- | --- | --- | --- |
| 是否展示学生名片 | 建议 | 控制是否出现名片模块 | 建议统一填：`是 / 否` |
| 学生名片简介 | 建议 | 名片模块文案 | 如果为空，脚本会自动给默认文案 |
| 学生标识 | 建议后续补 | 跨 offer 关联学生 | 未来如果要做到“按学生维度统一控制”，这个字段很关键 |

## 2. offer表：当前推荐填写规则

### 2.1 布尔字段统一规则
下面这些字段，不要随意混写，建议统一只用：

- `是`
- `否`

适用字段：
- `是否前台展示`
- `置顶标记`
- `是否展示学生名片`
- `最终选择标记`

### 2.2 排序权重规则
建议：
- 普通案例：`0`
- 想适度靠前：`10`
- 强主推案例：`50`
- 特别主推案例：`100`

排序逻辑当前为：
1. 先按 `置顶标记`
2. 再按 `排序权重` 倒序
3. 再按 `申请季 / 案例ID`

### 2.3 匿名展示名规则
建议统一格式：

- `A同学`
- `B同学`
- `C同学`

不要混用：
- `A`
- `A同学（金融）`
- `某同学`

除非你明确想让前台就这样展示。

### 2.4 国家/地区字段规则
建议统一枚举，避免筛选碎裂。

推荐值：
- `中国香港`
- `中国澳门`
- `中国内地`
- `新加坡`
- `英国`
- `美国`
- `法国`
- `意大利`
- `澳大利亚`
- `其他`

不要混用：
- `香港`
- `HK`
- `中国香港地区`

否则筛选会被拆成多个值。

- 还没有形成可运营维护的 key 体系

## 5. 当前脚本已做的兼容处理

为了让后台还没完全清洗完也能先运转，当前导出脚本已经做了兼容：

- `是否前台展示` 为空时：
  - 只要记录有基本内容，就默认导出
- `录取学校` 为空时：
  - 回退读 `录取院校`
- `本科院校` 为空时：
  - 回退读 `本科学校`
- `匿名展示名` 为空时：
  - 回退读 `学生姓名`
  - 再不行就用 `匿名同学`
- `学生名片简介` 为空时：
  - 自动给默认占位文案

**但这些都只是过渡兼容，不代表后台字段可以长期不整理。**

## 6. 推荐的下一步动作

### 优先级 A：先把 offer表 补到可控
建议优先补齐：
1. `是否前台展示`
2. `匿名展示名`
3. `录取国家/地区`
4. `排序权重`
5. `是否展示学生名片`
6. `学生名片简介`

### 优先级 B：初始化 页面配置表
建议按本文档第 3.3 节把 15 条基础 key 先建起来。

### 优先级 C：再做自动化
等你后台字段稳定后，再考虑：
- 自动导出
- 定时同步
- Webhook 触发
- 准实时前台更新

## 7. 当前标准操作流程

以后建议按这个流程使用：

### 7.1 改案例内容
在 `offer表` 修改：
- 是否展示
- 排序
- 学生名
- 国家/地区
- 名片
- 详情补充字段

### 7.2 改页面文案
在 `页面配置表` 修改对应的 `config_value`

### 7.3 导出前台数据
执行：

```bash
python3 feishu-bitable-sync/export_frontend_data.py
```

### 7.4 本地预览
在工作区执行：

```bash
python3 -m http.server 8000
```

打开：

```text
http://localhost:8000/ruc-suzhou-case-library/
```
